# Setting up the Catalyst kit in Figma Make

How to take everything in `make-kit/` and plug it into a Figma Make project so
generations are styled from Catalyst tokens and named for the Make → Design
System plugin to swap.

## 1. Create the Make project

In Figma Make, start a **new project from scratch** — not from a prompt. You
want an empty project so the kit can be the foundation, not something Make
overwrites with its own scaffolding.

## 2. Add the kit files

Add these 9 files to the Make project, preserving paths exactly:

```
tokens.css                       ← from make-kit/tokens.css
components.css                   ← from make-kit/components.css
components/Button.tsx
components/Badge.tsx
components/Avatar.tsx
components/TextAction.tsx
components/IconAction.tsx
components/Textfield.tsx
components/Input.tsx
components/index.ts
Stickersheet.tsx                 ← optional but useful (see step 5)
```

Don't rename anything — the `Component/variant/size` layer names depend on these
identifiers.

## 3. Wire tokens at the root

`tokens.css` only defines `:root` CSS variables — nothing styles until something
imports it. `components.css` adds the focus-visible / hover styles that can't
live in inline React style props (the blue focus outline, for example). In
Make's **entry file** (usually `App.tsx` or `index.tsx`), add both imports
once:

```tsx
import "./tokens.css";
import "./components.css";
```

Quickest sanity check: a primary button generated by Make should be **Catalyst
green** (`#00c983`). If it's grey/blue, the import isn't reaching the root.
Tab to it — it should show a blue accent focus ring; if not, `components.css`
isn't imported.

## 4. Set the project guidelines

Open Make's **project guidelines / system prompt** field (sometimes labeled
"guidelines", "knowledge", or "instructions" depending on the Make version) and
paste the entire contents of [`GUIDELINES.md`](./GUIDELINES.md).

This is the lever for naming. Without it, Make ignores your components and
writes raw `<div>`s; with it, every generation composes from `Button`, `Badge`,
etc., and emits `Component/variant/size` names that survive the copy into Figma.

## 5. Verify, don't trust

Make's exact layer-naming behavior is **not a documented contract**. Verify
once before you build on top:

1. Ask Make: *"Render the Stickersheet"* (or write `<Stickersheet />` in code).
2. Copy the result into a Figma file.
3. Open the **Layers panel** and confirm names read literally:
   - `Button/primary/large/labelOnly`
   - `Button/card/large/leadingIcon+label`
   - `Badge/info`
   - `Avatar/md/circle/initials/default/view`
   - `Textfield/edit/error`

If they don't, the kit's `data-name` strings aren't surviving Make's copy step.
Fix on the kit side and re-test. **Don't move past this step until it lines up
— everything downstream depends on it.**

## 6. Generate a real screen

Prompt Make with something domain-real, e.g. *"Build a sign-in form for
CATAPA: email, password, submit, forgot password link."* You should get a
screen composed entirely from kit components, no raw divs. Copy into Figma →
run the **Make → Design System** plugin → swap.

## 7. When Make doesn't preserve names

Sometimes Figma Make pastes layers as `Frame 1234` / `Group 5` instead of the
kit's `Component/variant/size/...` names — its handling of `data-name` isn't a
guarantee. In that case:

1. Select the pasted layers in Figma.
2. Run the plugin and press **Auto-name** before pressing **Swap**.
3. The plugin walks the selection, infers the right name from each layer's
   colour/structure (rules in [`RENAME-RULES.md`](./RENAME-RULES.md)), and
   re-scans so the suggestion list lights up.

You can usually skip this if step 5 verified your generations come through with
the right names; treat it as the fallback for when Make's output drifts.

## When you update later

The pieces have clear ownership:

| Want to change                  | Edit                          | Then                                                                |
| ------------------------------- | ----------------------------- | ------------------------------------------------------------------- |
| A color, spacing, shadow        | `tokens/*.json`               | `node make-kit/build-tokens.js` → re-upload `tokens.css` to Make    |
| Component behavior or naming    | `make-kit/components/*.tsx`   | Re-upload that file to Make                                         |
| What Make should/shouldn't do   | `make-kit/GUIDELINES.md`      | Paste new content into Make's guidelines field                     |
| A variant matrix                | `components/<name>.json` + the make-kit component + the Figma DS library | Three places — all should match |

## Caveats

- **Make's file-loading and naming behavior can vary by Make version.** Step 5
  is the verification. If it doesn't match expectations, the fix is on the kit
  side (`data-name` strings) or Make's prompt/guidelines — not the plugin.
- **The kit is incomplete relative to your full DS.** Only 7 components have
  JSON specs today: Button, Badge, Avatar, TextAction, IconAction, Textfield,
  Input. Your real library has Cards, Banners, Carousel, etc. When you scale,
  the pattern is identical: add a `*.json` spec → add a `.tsx` component → add
  its naming convention to `GUIDELINES.md`.
- **The Figma destination library still needs the matching component sets.**
  The plugin swaps Make output **into** real DS components — those must exist in
  Figma as component sets with variant properties whose values match the kit's
  variant names. The Button set already does; the other 6 are TBD.
